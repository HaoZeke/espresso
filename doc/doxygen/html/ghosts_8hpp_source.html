23 /** \file ghosts.hpp Ghost particles and particle exchange.
24 
25 In this file you find everything concerning the exchange of
26 particle data (particles, ghosts, positions and forces) for short
27 range interactions between the spacial domains of neighbouring
28 nodes.
29 
30 <h2> How does this work </h2>
31 The ghost communication transfers data from cells on one node to cells on another node during
32 the integration process. Note that data can only be transferred from one cell to one other, and
33 the contents of the other cell will be overwritten. This communication is invoked by the integrator,
34 at four different times in an integration step. These steps are reflected in \ref cell_structure structure,
35 since they are treated differently by different cell systems:
36 <ul>
37 <li> ghost_cells are used to transfer the cell sizes, i.e., make sure that for all later transfers
38  the target cell has the same size as the source cell.
39 <li> exchange_ghosts sets up all information on the ghosts that is necessary. Normally transfers the
40  (shifted) position and particle properties.
41 <li> update_ghosts is used to update the particle properties if no particles have been moved between cells.
42  Therefore only the positions are transferred, otherwise this normally looks pretty much like the
43  exchange_ghosts.
44 <li> collect_ghost_forces finally is used to transfer back forces that were exerted on a ghost particle. They
45  are simply added again to the force of the real particle. The communication process is therefore inverted
46  with respect to exchange_ghosts and update_ghosts, i.e., sending is replaced by receiving and the other way
47  round.
48 </ul>
49 The particle data that has to be transferred, and especially from where to where, heavily depends on the cell system.
50 In Espresso, this is abstracted in form of ghost communicators (\ref GhostCommunicator) and ghost communications
51 (\ref GhostCommunication). The ghost communicators represent the four communications above and consist of the data to
52 transfer (which is determined by their type) and a list of ghost communications.
53 The data types are described by the particle data classes
54 <ul>
55 <li> GHOSTTRANS_PROPRTS transfers the \ref ParticleProperties
56 <li> GHOSTTRANS_POSITION transfers the \ref ParticlePosition
57 <li> GHOSTTRANS_POSSHFTD is no transfer class, but marks that an additional vector, the shift, has to be
58  added to the positions. Can only be used together with GHOSTTRANS_POSITION.
59 <li> GHOSTTRANS_MOMENTUM transfers the \ref ParticleMomentum
60 <li> GHOSTTRANS_FORCE transfers the \ref ParticleForce
61 <li> GHOSTTRANS_PARTNUM transfers the cell sizes
62 </ul>
63 Each ghost communication describes a single communication of the local with another node (or all other nodes). The data
64 transferred can be any number of cells, there are five communication types:
65 <ul>
66 <li> GHOST_SEND sends data to one other node
67 <li> GHOST_RECV recvs data from one other node. In the case of forces, they are added up, everything else is overwritten
68 <li> GHOST_BCST sends data to all nodes
69 <li> GHOST_RDCE recvs data from all nodes. In the case of forces, they are added up, everything else is overwritten
70 <li> GHOST_LOCL transfer data from a local cell to another local cell. In this case, the first half of the cells are the sending cells,
71  the other half are the corresponding receivers.
72 </ul>
73 Note that for the first four communications you have to make sure that when one node sends to another, that the sender has GHOST_SEND and the
74 receiver GHOST_RECV. In the case of GHOST_BCST rsp. GHOST_RDCE, all nodes have to have the same communication type and the same master
75 sender/receiver (just like the MPI commands).
76 
77 A special topic are GHOST_PREFETCH and GHOST_PSTSTORE. For example, if all nodes broadcast to the other, the naive implementation will be that
78 n_nodes times a GHOST_BCST is done with different master nodes. But this means that each time n_nodes-1 nodes wait for the master to construct
79 its send buffer. Therefore there is the prefetch flag which can be set on a pair of recv/send operations. If the ghost communication reaches a
80 recv operation with prefetch, the next send operation (which must have the prefetch set!!) is searched and the send buffer already created.
81 When sending, this precreated send buffer is used. In the scenario above, all nodes create the send buffers simultaneously in the first
82 communication step, thereby reducing the latency a little bit. The pststore is similar and postpones the write back of received data until a
83 send operation (with a precreated send buffer) is finished.
84 
85 The ghost communicators are created in the init routines of the cell systems, therefore have a look at \ref dd_topology_init or
86 \ref nsq_topology_init for further details.
87 */
